### Upgrade 8: Cross-Encoder Reranking (Sección 9, 51 de la guía)

**Concepto:** Después del retrieval (FAISS + BM25 + RRF), los top-10 resultados pasan por un segundo modelo (cross-encoder) que evalúa cada par query↔chunk JUNTOS. Es más lento pero mucho más preciso. Reordena los resultados — a veces el chunk #8 es más relevante que el #1.

**Implementación en `rag_search.py`:**

1. Instalar: `pip install sentence-transformers` (ya lo tenemos).
2. Cargar un cross-encoder local: `cross-encoder/ms-marco-MiniLM-L-6-v2` (pequeño, rápido, gratuito).
3. Después del RRF fusion (que devuelve top-10), pasar cada par (query, chunk.raw_text) al cross-encoder.
4. El cross-encoder devuelve un score de relevancia por par.
5. Reordenar por ese score y devolver top-5.

```python
from sentence_transformers import CrossEncoder

reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

def rerank(query: str, chunks: list[dict], top_k: int = 5) -> list[dict]:
    pairs = [(query, c["raw_text"]) for c in chunks]
    scores = reranker.predict(pairs)
    for chunk, score in zip(chunks, scores):
        chunk["rerank_score"] = float(score)
    chunks.sort(key=lambda x: x["rerank_score"], reverse=True)
    return chunks[:top_k]
```

6. En el frontend, mostrar `rerank_score` junto con `rrf_score` para transparencia.
7. En la evaluación, agregar un cuarto modo: "hybrid+rerank" y comparar Recall@5 contra hybrid sin rerank.

**Diagnóstico de cuándo activarlo:** Si en la evaluación ves que el Recall@5 es alto (los docs correctos ESTÁN en top-10) pero las respuestas del LLM son malas, el problema es el ORDERING — reranking lo resuelve. Si el Recall@5 es bajo, el problema es el retrieval y reranking no ayuda.

### Upgrade 9: Query Cache (Sección 6, 11 de la guía)

**Concepto:** Queries repetidas como "how to reset VPN" o "password expired" no necesitan pasar por todo el pipeline cada vez. Cacheas la respuesta y la sirves en ~5ms en vez de ~2 segundos.

**Implementación en `rag_search.py`:**

1. Crear un dict en memoria: `query_cache = {}`.
2. Key: el query normalizado (lowercase, stripped).
3. Value: la respuesta completa + timestamp.
4. Antes de buscar, checar el cache. Si existe y tiene menos de 24 horas, devolver la respuesta cacheada.
5. Agregar header `X-Cache: HIT` o `X-Cache: MISS` en la respuesta HTTP.
6. Endpoint `GET /cache/stats` que muestre: tamaño del cache, hit rate, queries más frecuentes.
7. Endpoint `POST /cache/clear` para limpiar el cache (útil después de re-indexar).

```python
import time
from collections import OrderedDict

class QueryCache:
    def __init__(self, max_size: int = 500, ttl_seconds: int = 86400):
        self.cache = OrderedDict()
        self.max_size = max_size
        self.ttl = ttl_seconds
        self.hits = 0
        self.misses = 0

    def get(self, query: str):
        key = query.lower().strip()
        if key in self.cache:
            entry = self.cache[key]
            if time.time() - entry["timestamp"] < self.ttl:
                self.hits += 1
                self.cache.move_to_end(key)  # LRU
                return entry["response"]
            else:
                del self.cache[key]
        self.misses += 1
        return None

    def set(self, query: str, response: dict):
        key = query.lower().strip()
        if len(self.cache) >= self.max_size:
            self.cache.popitem(last=False)  # remove oldest
        self.cache[key] = {"response": response, "timestamp": time.time()}

    def stats(self):
        total = self.hits + self.misses
        return {
            "size": len(self.cache),
            "hits": self.hits,
            "misses": self.misses,
            "hit_rate": round(self.hits / total, 3) if total > 0 else 0,
        }
```

8. Invalidar cache automáticamente cuando se corre `build_rag.py` (guardar un timestamp de último build en `rag_artifacts/last_build.json` y comparar).

### Upgrade 10: Input Guardrails (Sección 37 de la guía — Defense in Depth)

**Concepto:** Antes de procesar un query, validar que no sea: prompt injection, contenido ofensivo, o fuera de scope.

**Implementación en `rag_search.py`:**

1. **Sanitización básica:** Quitar caracteres especiales sospechosos, limitar longitud del query a 500 chars.

2. **Detección de prompt injection:** Buscar patrones conocidos:
```python
INJECTION_PATTERNS = [
    r"ignore\s+(previous|above|all)\s+instructions",
    r"you\s+are\s+now\s+",
    r"system\s*:\s*",
    r"<\|.*?\|>",
    r"```.*?```",
    r"forget\s+(everything|what|your)",
    r"new\s+instructions",
    r"act\s+as\s+",
    r"pretend\s+(you|to)\s+",
]

def check_injection(query: str) -> bool:
    for pattern in INJECTION_PATTERNS:
        if re.search(pattern, query, re.IGNORECASE):
            return True
    return False
```

3. **Delimitadores:** Cuando construyas el prompt para el LLM, separar claramente user content de system content:
```
=== SYSTEM INSTRUCTIONS (DO NOT MODIFY) ===
{system_prompt}
=== END SYSTEM ===

=== KB DOCUMENTS ===
{context}
=== END DOCUMENTS ===

=== USER QUESTION ===
{query}
=== END USER ===
```

4. **Output validation:** Después de que el LLM responde, verificar:
   - No contiene PII patterns (SSN: `\d{3}-\d{2}-\d{4}`, credit card: `\d{4}[\s-]?\d{4}[\s-]?\d{4}[\s-]?\d{4}`)
   - El JSON es válido
   - Confidence es un número entre 0 y 1
   - Sources referencian KB numbers que realmente existen en el index

5. Loguear todos los intentos bloqueados a `data/logs/blocked_queries.jsonl`.

### Upgrade 13: Confidence-Based Degradación Elegante (Sección 11 de la guía — Fallbacks)

**Concepto:** Cuando el sistema no está seguro, debe degradar elegantemente en vez de dar una respuesta mala.

**Implementación en `rag_search.py`:**

Tres niveles según el score de confianza:

```python
def determine_response_tier(confidence: float, top_rrf_score: float) -> str:
    if confidence >= 0.8 and top_rrf_score >= 0.025:
        return "full_response"       # Respuesta completa con message + notes
    elif confidence >= 0.5:
        return "cautious_response"   # Respuesta con disclaimer + KB links
    else:
        return "escalation"          # No responder, sugerir escalación
```

- **full_response:** Respuesta normal con suggested_message, notes, y prompt.
- **cautious_response:** "I found some possibly relevant articles but I'm not confident. Please review: [KB links]. Consider escalating if these don't help."
- **escalation:** "I don't have enough information to answer this. Recommended escalation to: [escalation_team from KB metadata]."

En el frontend, mostrar un badge de color: verde (full), amarillo (cautious), rojo (escalation).

### Upgrade 14: Query Enhancement — Multi-Query (Sección 7 de la guía)

**Concepto:** Un solo query puede ser ambiguo. Generar 2-3 reformulaciones y buscar con todas, luego fusionar resultados.

**Implementación en `rag_search.py`:**

Solo si Ollama está corriendo. Si no, skip.

```python
def generate_query_variants(query: str) -> list[str]:
    """Genera 2-3 reformulaciones del query original."""
    prompt = f"""Generate 3 different ways to search for information about this IT support issue.
Each variant should use different words but mean the same thing.
Return ONLY a JSON array of strings, nothing else.

Original query: {query}"""

    # Call Ollama
    response = requests.post(f"{OLLAMA_BASE_URL}/api/generate",
        json={"model": "llama3", "prompt": prompt, "format": "json",
              "stream": False, "options": {"temperature": 0.3, "num_predict": 150}},
        timeout=30)

    variants = json.loads(response.json()["response"])
    return [query] + variants[:2]  # original + 2 variants max
```

Luego buscar con CADA variante, y fusionar todos los resultados con RRF (cada variante es un "retriever" adicional).

**Flag para activar/desactivar:** `ENABLE_MULTI_QUERY = True` en config. Desactivar para queries donde la velocidad importa más que la cobertura.

### Upgrade 15: Metadata Filters en Retrieval (Sección 8 de la guía — Routing)

**Concepto:** `rag_search.py` ya detecta el sistema (Five9, Okta, etc.) y el segmento (retail/corporate). Usar eso para FILTRAR resultados ANTES del ranking.

**Implementación:**

Actualmente `rag_search.py` detecta el sistema. El upgrade es:
1. Si el sistema fue detectado con alta confianza, filtrar los resultados post-RRF para solo incluir chunks del mismo sistema.
2. Si fue detectado con baja confianza, NO filtrar (dejar que el RRF score decida).
3. Agregar al endpoint: `POST /query` acepta campo opcional `system_filter` y `segment_filter`.

```python
def apply_metadata_filters(chunks: list[dict], system: str = None,
                           segment: str = None, confidence: float = 0.0) -> list[dict]:
    if not system or confidence < 0.7:
        return chunks  # no filtrar si baja confianza

    filtered = [c for c in chunks if system.lower() in c.get("system", "").lower()]

    # Si filtramos demasiado, devolver sin filtro
    if len(filtered) < 2:
        return chunks

    return filtered
```

### Upgrade 16: Prompt Versioning (Sección 36 de la guía — Experiment Tracking)

**Concepto:** Versionar los prompts para poder comparar resultados entre versiones.

**Implementación:**

1. Crear carpeta `prompts/` con archivos por versión:
   - `prompts/system_v1.txt` — prompt original
   - `prompts/system_v2.txt` — con suggested_message/notes
   - `prompts/system_v3.txt` — con guardrails mejorados

2. En `config.py`: `ACTIVE_PROMPT_VERSION = "v2"`

3. En `rag_search.py`: cargar el prompt desde el archivo según la versión activa.

4. En session traces: loguear qué versión de prompt se usó.

5. En `evaluate.py`: poder evaluar con diferentes versiones de prompt:
   ```bash
   python evaluate.py --prompt-version v1
   python evaluate.py --prompt-version v2
   ```

| Active RAG / CRAG / Self-RAG (10) | Requiere múltiples llamadas al LLM por query (grading de relevancia, grading de alucinación). Muy lento con modelos locales. Candidato para cuando tengamos Azure OpenAI. |
| LangGraph state machine (10) | Overkill para este pipeline. El flujo es lineal con branching simple (confidence tiers). Si el proyecto crece a multi-step agents, ahí sí. |
| Computer Vision / YOLO | No aplica — no hay imágenes en este proyecto. |
| Kubernetes / microservicios | Es un POC local. Cuando se despliegue en Azure App Service, sigue siendo un monolito simple. |
| Azure AI Search migration | Requiere acceso a Azure y presupuesto. Es Fase 2. |
