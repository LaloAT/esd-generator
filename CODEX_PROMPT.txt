## What to upgrade

The current system is semi-manual: the agent pastes a ticket, the system suggests KBs and generates a prompt, then the agent copies that prompt into a Custom GPT. The upgrades make it more automated and add quality measurement.

### Upgrade 1: Add BM25 hybrid search to `build_rag.py`

**Current:** FAISS semantic search only (with TF-IDF fallback).
**Upgrade:** Keep FAISS semantic search AND add a BM25 index. At search time, run both and fuse with Reciprocal Rank Fusion (RRF).

Changes to `build_rag.py`:
- After building FAISS index, also build a BM25Okapi index from the raw text of each KBRecord.
- Tokenize: lowercase, split by `\w+`, remove English stopwords.
- Serialize BM25 index to `rag_artifacts/kb_bm25.pkl` using pickle.
- Add `rank-bm25>=0.2.2` to dependencies.

Changes to `rag_search.py`:
- On startup, load both FAISS index AND BM25 pickle.
- When searching, run both:
  - FAISS: embed query with same model, search top-10.
  - BM25: tokenize query, get top-10 scores.
- Fuse with RRF: `score(doc) = sum(1/(60 + rank_i(doc)))` for each retriever where the doc appears.
- Return top-5 after fusion.
- Add `matched_by` field to results: ["semantic"], ["bm25"], or ["semantic", "bm25"].

**Why:** Semantic search misses exact matches (KB numbers, error codes like "0x80070005", product names). BM25 catches those. Together they cover both natural language and exact keyword queries.

### Upgrade 2: Add semantic chunking to `build_rag.py`

**Current:** Articles are likely indexed as whole documents or with basic splitting.
**Upgrade:** Add semantic chunking that splits articles where the embedding similarity between consecutive sentences drops.

Changes to `build_rag.py`:
- Before embedding, split each article body into sentences (use nltk.sent_tokenize).
- Embed all sentences with the same model.
- Calculate cosine similarity between consecutive sentences.
- Cut where similarity falls below the 25th percentile.
- If a resulting chunk exceeds 800 chars, subdivide by size.
- Enrich each chunk before embedding: prepend article title, kb_number, system, category.
- Also extract factual chunks: scan for phone numbers, URLs, emails, error codes with regex. Put them in dedicated chunks tagged as `chunk_type: "factual"`.
- Add `nltk>=3.8` to dependencies.

### Upgrade 4: Direct LLM generation (bypass manual GPT copy-paste)

**Current:** System generates a prompt that the agent copies into the Custom GPT manually.
**Upgrade:** Add an OPTIONAL mode where the system calls a local LLM (Ollama) directly and generates the full response including suggested message, notes, and the fallback prompt for manual GPT.

Changes to `rag_search.py`:
- Add a new endpoint: `POST /generate` that takes the query + retrieved chunks and calls Ollama directly.
- Uses this system prompt (incorporates the style from `GPT_INSTRUCTIONS.txt` and `ESD_formatted_notes_and_messages.txt`):

```
You are Lalo's AI assistant for Oportun ESD L1 support.

For every ticket, provide ALL of the following in JSON:

1. "answer": Technical answer for the agent. Cite KB numbers. Include troubleshooting steps.

2. "suggested_message": Professional, empathetic message to send to the end user via Slack.
   - Do NOT include KB numbers (users don't see those)
   - Use simple language, step-by-step if applicable
   - Match the ESD team's tone (professional but warm)
   - End with "If you continue to experience issues, please let us know."
   - Keep under 150 words

3. "suggested_notes": Work notes for ServiceNow ticket.
   - Technical and concise, bullet points
   - Include KB numbers referenced
   - Include: what was found, what was suggested, resolution status
   - End with "Source: LLM-ESD RAG"

4. "llm_prompt": A ready-to-paste prompt for the Custom GPT. Include:
   - The relevant KB article content (actual text, not just numbers)
   - The user's original question/issue
   - Instructions for the output format (Slack message, email, work notes, etc.)
   - Note about tone (professional, empathetic, ESD style)

RULES:
- ONLY use information from the provided KB articles.
- If the articles don't contain the answer: say so in all fields, recommend escalation.
- NEVER invent phone numbers, URLs, IDs, or any factual data.
- Always cite KB numbers in answer and suggested_notes (NOT in suggested_message).

RESPOND IN VALID JSON ONLY.
```

- If Ollama is not running, the endpoint returns an error message suggesting to use the manual prompt method instead.
- The existing prompt-generation flow continues to work as before.

Changes to `rag_frontend.html`:
- Add a new section after "GENERATED GPT PROMPT": **"5. AI-GENERATED RESPONSE"**
- A button "Generate with Local LLM" that calls `POST /generate`.
- If successful, show four collapsible sections: Answer, Suggested Message (with Copy button), Suggested Notes (with Copy button), LLM Prompt (with Copy button).
- If the server returns an error (Ollama not running), show a message: "Local LLM not available. Use the GPT Prompt above instead."
- The existing flow (Parse Ticket → KB Suggestions → GPT Prompt) remains untouched.

### Upgrade 5: Incremental KB updates

**Current:** Full rebuild every time (`build_rag.py` processes all 580 files).
**Upgrade:** Only re-process new or modified files.

Changes to `build_rag.py`:
- On each run, compute MD5 hash of each `.txt` file.
- Store hashes in `rag_artifacts/file_hashes.json`.
- Compare with previous hashes: classify as NEW, MODIFIED, UNCHANGED, DELETED.
- Only parse/embed NEW and MODIFIED articles.
- Rebuild FAISS index from full chunk set (FAISS doesn't support in-place deletion).
- Rebuild BM25 index.
- Add `--full` flag to force full rebuild.
- Print summary: "3 new, 7 modified, 1 deleted, 569 unchanged (skipped)".

### Upgrade 6: Evaluation and golden dataset

**Current:** No way to measure retrieval quality.
**Upgrade:** Add evaluation scripts and a golden dataset.

New file `evaluate.py`:
- Load `golden_dataset.json` (query + expected KB number pairs).
- Run each query through the retriever in three modes: semantic-only, bm25-only, hybrid.
- Calculate Recall@1, Recall@5, MRR for each mode.
- Print comparison table.
- Save results to `rag_artifacts/eval_results.json`.

New file `golden_dataset.json`:
- Auto-generate a template from existing articles (title + symptoms → suggested query).
- User edits queries to be natural language ("agent cant login five9" instead of "How to Reset Five9 Agent Password").
- Support importing real queries from `ESD_formatted_notes_and_messages.txt` — parse out the original issues/questions from the examples.

New file `compare_models.py`:
- Compare all-MiniLM-L6-v2 (384 dims) vs all-mpnet-base-v2 (768 dims) on the golden dataset.
- Print Recall@5 and MRR for each.
- Recommend which model to use.
